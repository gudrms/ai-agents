# Project Start Harness

이 스킬은 Codex가 프로젝트나 기능 작업을 시작할 때 요구사항, 범위, 검증 방법을 놓치지 않도록 돕는 전역 작업 하네스다.

## 핵심 원칙

구현 전에 가벼운 기준 문서를 먼저 확인하거나 만든다.

우선순위는 다음과 같다.

1. 기존 프로젝트 문서
2. Spec Kit 산출물
3. `docs/TASK_CONTEXT.md`
4. 작은 작업이면 채팅 내 짧은 계획

이미 충분한 계획 문서가 있으면 중복 문서를 만들지 않는다.

## 작업 시작 체크리스트

작은 수정이 아닌 작업을 시작할 때는 먼저 다음을 확인한다.

- `AGENTS.md`
- `README.md`
- `package.json`, `pyproject.toml`, `docker-compose.yml` 같은 실행/빌드 설정
- `specs/`, `.specify/`, `.agents/skills/`
- `docs/TASK_CONTEXT.md`
- 현재 git 변경사항
- 관련 테스트 파일

단순 버그 수정이나 10분 이내의 작은 변경에는 과한 절차를 적용하지 않는다.

## Spec Kit 사용 기준

다음 작업에는 Spec Kit을 우선 사용한다.

- 새 프로젝트
- 새 기능
- 과제 또는 포트폴리오 프로젝트
- 요구사항이 애매한 작업
- 여러 단계로 나뉘는 리팩터링
- 다음 세션에서 이어받을 가능성이 큰 작업
- 검증 기준이 중요한 작업

프로젝트에 Spec Kit이 없고 작업이 충분히 크면 다음 명령을 제안한다.

```powershell
specify init --here --ai codex --ai-skills
```

사용자가 Spec Kit 사용을 원하거나 이미 Spec Kit이 초기화되어 있으면 다음 흐름을 따른다.

1. constitution
2. specify
3. clarify
4. plan
5. tasks
6. implement

## TASK_CONTEXT 사용 기준

다음 경우에는 `docs/TASK_CONTEXT.md`를 사용한다.

- Spec Kit이 설치되어 있지 않다.
- 프로젝트가 작다.
- 사용자가 가벼운 하네스를 원한다.
- 세션이 길어지고 있다.
- 다음 세션에서 이어받아야 한다.
- 작업 중 목표나 범위가 바뀌었다.

`docs/TASK_CONTEXT.md`가 없고 작업이 장기화될 가능성이 있으면 생성하거나 생성 제안을 한다.

권장 구조는 다음과 같다.

```md

-

-
```

## 계획 수립 규칙

작은 수정이 아닌 작업에서 파일을 수정하기 전에는 다음을 정리한다.

- 목표
- 포함 범위
- 제외 범위
- 현재 가정
- 수정 가능성이 있는 파일
- 검증 방법

다음 영역의 불확실성이 있으면 구현 전에 짧게 질문한다.

- 아키텍처
- 데이터 손실 가능성
- 보안
- 인증/인가
- 결제/과금
- 사용자에게 보이는 동작
- DB 스키마
- 외부 API 계약

## 구현 규칙

구현 중에는 다음을 지킨다.

- 현재 목표와 직접 관련된 변경만 한다.
- 사용자가 만든 기존 변경사항을 되돌리지 않는다.
- 범위가 바뀌면 계획 문서를 갱신한다.
- 동작이 바뀌면 테스트를 추가하거나 갱신한다.
- 중요한 설계 결정은 `docs/TASK_CONTEXT.md`, Spec Kit 문서, 또는 결정 로그에 남긴다.
- 새 의존성 추가, DB 변경, 인증 변경, 배포 설정 변경은 이유를 명확히 한다.

## 완료 전 체크리스트

최종 응답 전 다음을 확인한다.

- 구현이 현재 목표와 일치하는가
- 포함 범위와 제외 범위를 지켰는가
- 테스트 또는 수동 검증을 실행했는가
- 실행하지 못한 검증이 있다면 이유를 설명할 수 있는가
- 남은 리스크가 있는가
- 작업이 이어질 경우 `docs/TASK_CONTEXT.md` 또는 Spec Kit 산출물이 갱신되었는가

## 작업 규모별 기본 흐름

### 작은 버그 수정

1. 관련 파일만 확인한다.
2. 최소 변경으로 수정한다.
3. 가능한 가장 좁은 검증을 실행한다.
4. 변경 내용과 검증 결과를 요약한다.

### 기능 개발

1. 기존 문서와 하네스를 확인한다.
2. 요구사항, 범위, 제외 범위, 검증 방법을 정리한다.
3. 필요한 경우 `docs/TASK_CONTEXT.md` 또는 Spec Kit 문서를 갱신한다.
4. 구현을 작은 단위로 나눈다.
5. 구현 후 테스트 또는 수동 검증을 실행한다.
6. 이어받을 상태를 남긴다.

### 새 프로젝트

1. Spec Kit 사용 가능 여부를 확인한다.
2. 가능하면 Spec Kit으로 시작한다.
3. 그렇지 않으면 `AGENTS.md`와 `docs/TASK_CONTEXT.md`를 만든다.
4. 성공 기준과 검증 방법을 정한 뒤 구현한다.

## bkit과의 관계

Claude 환경에서 bkit 문서가 있는 프로젝트라면 bkit 문서를 우선 존중한다.

- bkit PDCA 문서가 있으면 그것을 상위 계획으로 본다.
- Codex에서는 Spec Kit 또는 `TASK_CONTEXT.md`를 기능 단위 실행 하네스로 사용한다.
- 같은 목적의 계획 문서를 중복 생성하지 않는다.
- bkit과 Spec Kit 문서가 모두 있으면 다음 순서를 따른다.

1. bkit PRD/PDCA 상위 문서
2. Spec Kit feature spec/plan/tasks
3. `docs/TASK_CONTEXT.md`
4. 실제 코드와 테스트

## 최종 응답 규칙

작업 완료 후에는 필요한 경우 다음을 짧게 보고한다.

- 변경한 파일
- 핵심 변경 내용
- 실행한 검증
- 생략한 검증과 이유
- 남은 리스크 또는 다음 액션
